# Comprehensive Guide to Setting Up the "ProcessAI" Project

This guide provides a step-by-step walkthrough for configuring the necessary infrastructure on Google Cloud and Firebase, deploying the backend services, and connecting the Next.js frontend for the ProcessAI application.

## 1. Google Cloud and Firebase Project Setup

### 1.1. Create a Google Cloud Project

1.  Go to the [Google Cloud Console](https://console.cloud.google.com/).
2.  Click on the project selector at the top of the page and then click on **"New Project"**.
3.  Enter a project name (e.g., "ProcessAI") and click **"Create"**. Make a note of the **Project ID**, as you will need it later.

### 1.2. Enable Necessary APIs

1.  In the Google Cloud Console, navigate to the **"APIs & Services" > "Library"** section.
2.  Search for and enable the following APIs for your project:
    *   Cloud Firestore API
    *   Cloud Storage
    *   Cloud Functions API
    *   Cloud Vision AI API
    *   Vertex AI API
    *   Generative Language API
    *   Cloud Build API (usually enabled by default, but good to check)
    *   Identity and Access Management (IAM) API
    *   Cloud Run Admin API

### 1.3. Create and Configure a Firebase Project

1.  Go to the [Firebase Console](https://console.firebase.google.com/).
2.  Click on **"Add project"** and select your newly created Google Cloud project.
3.  Follow the on-screen instructions to add Firebase to your project.

## 2. Firebase Services Configuration

### 2.1. Firestore

#### 2.1.1. Create Firestore Database

1.  In the Firebase Console, go to the **"Firestore Database"** section.
2.  Click on **"Create database"**.
3.  Choose to start in **"Production mode"** and select a location for your database.
4.  Click **"Enable"**.

#### 2.1.2. Suggested Data Structure

Based on your application's needs, here is a suggested Firestore collection structure:

*   **`documents` collection:**
    *   `documentId` (document ID)
        *   `fileName`: `string` (e.g., "annual-report-2023.pdf")
        *   `status`: `string` ("uploading", "processing", "processed", "error")
        *   `internalStatus`: `string` (e.g., "batch_document_ai_started")
        *   `createdAt`: `timestamp`
        *   `updatedAt`: `timestamp`
        *   `summary`: `string` (generated by the summarization flow)
        *   `gcsInputUri`: `string` (path to the raw PDF in Cloud Storage)
        *   `gcsOutputUri`: `string` (path to the processed output)

*   **`chats` collection:**
    *   `chatId` (chat session ID)
        *   `documentId`: `string` (reference to the document being discussed)
        *   `userId`: `string` (reference to the user)
        *   `createdAt`: `timestamp`
        *   `messages`: `array` of objects
            *   `sender`: `string` ("user" or "ai")
            *   `content`: `string`
            *   `timestamp`: `timestamp`

#### 2.1.3. Firestore Security Rules

Go to the **"Rules"** tab in the Firestore section of the Firebase Console and use the following rules as a starting point. These rules assume you are using Firebase Authentication.

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Users can only access their own documents and chats.
    match /documents/{documentId} {
      allow read, write: if request.auth != null && request.auth.uid == resource.data.userId;
    }

    match /chats/{chatId} {
      allow read, write: if request.auth != null && request.auth.uid == resource.data.userId;

      // Users can only create chats for themselves
      allow create: if request.auth != null && request.resource.data.userId == request.auth.uid;

      // Allow users to read and write their own messages
      match /messages/{messageId} {
        allow read, write: if request.auth != null && get(/databases/$(database)/documents/chats/$(chatId)).data.userId == request.auth.uid;
      }
    }
  }
}
```

### 2.2. Cloud Storage

#### 2.2.1. Create Storage Buckets

By default, Firebase creates a default bucket for you. You can use it or create new ones. For this project, we'll use two folders inside the default bucket: `uploads/` for raw PDFs and `results/` for the processed output from Document AI.

#### 2.2.2. Cloud Storage Security Rules

Go to the **"Rules"** tab in the Storage section of the Firebase Console and use the following rules.

```
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    // Allow authenticated users to upload to the 'uploads/{userId}/{documentId}' path.
    match /uploads/{userId}/{documentId}/{fileName} {
      allow write: if request.auth != null && request.auth.uid == userId;
    }

    // Allow read access to the processed results for the user who owns the document.
    match /results/{documentId}/{allPaths=**} {
      allow read: if request.auth != null && exists(/databases/$(database)/documents/documents/$(documentId)) &&
                   get(/databases/$(database)/documents/documents/$(documentId)).data.userId == request.auth.uid;
    }
  }
}
```

#### 2.3.2. Deploying Functions and Environment Variables

1.  **Set Environment Variables:**
    Store sensitive information like API keys and project IDs as environment variables in your functions. You can set them using the Firebase CLI:
    ```bash
    firebase functions:config:set gcp.project="YOUR_PROJECT_ID"
    firebase functions:config:set gcp.location="YOUR_PROJECT_LOCATION"
    # Add other necessary variables
    ```
2.  **Deploy Functions:**
    ```bash
    firebase deploy --only functions
    ```

## 3. Google Cloud AI and Genkit Integration

### 3.1. Service Account Credentials

Your Cloud Functions will need permissions to access other Google Cloud services. The default App Engine service account, which functions use by default, has broad permissions. For a more secure setup, create a dedicated service account:

1.  Go to **"IAM & Admin" > "Service Accounts"** in the Cloud Console.
2.  Click **"Create Service Account"**.
3.  Give it a name (e.g., "processai-functions-runner") and an ID.
4.  Grant it the following roles:
    *   **Cloud Functions Invoker**
    *   **Cloud Vision AI User**
    *   **Vertex AI User**
    *   **Firestore Service Agent**
    *   **Storage Object Admin**
5.  When deploying your functions, you can specify this service account.

### 3.2. Initializing Genkit in Cloud Functions

In your `src/ai/genkit.config.ts` file, you will configure Genkit to use the Google AI provider.

```typescript
import { googleAI } from 'genkitx-googleai';
import { configureGenkit } from '@genkit-ai/core';

export const initializeGenkit = () => {
  configureGenkit({
    plugins: [
      googleAI(),
    ],
    logLevel: 'debug',
    enableTracingAndMetrics: true,
  });
};
```

You then call `initializeGenkit()` in your `src/functions/index.ts` as shown above.

### 3.3. Configuring Vertex AI Vector Search

1.  **Create an Index:**
    *   Go to the **Vertex AI > Vector Search** section in the Cloud Console.
    *   Click **"Create Index"**.
    *   Give your index a name.
    *   Upload a small sample of your embedding data (even a dummy file with the correct dimensions) to let Vertex AI configure the index. The `text-embedding-004` model produces 768-dimensional embeddings.
    *   Choose the "Batch" update method.
2.  **Create an Index Endpoint:**
    *   After the index is created, create an **"Index Endpoint"**.
    *   Deploy your index to this endpoint. This will make it available for online queries.

## 5. Security Considerations

*   **API Keys:** Never hardcode API keys in your client-side code. Use Cloud Functions as a backend to securely call Google Cloud services. Manage keys using environment variables or a secret manager like Google Secret Manager.
*   **Firebase Authentication:** Use Firebase Authentication to secure your application. It provides easy-to-use SDKs to authenticate users and integrates seamlessly with Firestore and Cloud Storage security rules.
*   **Principle of Least Privilege:** When configuring IAM roles for service accounts, grant only the permissions that are absolutely necessary.
